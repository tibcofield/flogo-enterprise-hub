## The Problem

SAP S/4HANA modernization programs are among the most complex integration challenges an enterprise can undertake. Migrating from ECC — or running a parallel-landscape hybrid — means dozens of integration flows are in constant motion: order-to-cash, procure-to-pay, finance close, logistics execution, and more. Each of those flows is wired together by an **integration middleware layer** (TIBCO BusinessWorks 6 in our case), connecting SAP to downstream systems like payment processors, warehouse management, fraud screening, and data lakes.

When something goes wrong in this layer — a database pool exhaustion during a batch sync, an upstream REST 500 from a partner API, a data quality mismatch from an IDoc transformation — the integration platform fires an error event. That event gets routed to **ServiceNow as a new incident ticket**.

The problem is what happens next. A single root cause — say, the S/4HANA database is slow during a month-end close batch — can cascade across dozens of integration pods simultaneously. The on-call engineer opens their queue and sees this:

```
INC0001001  [BW-JDBC-100014] OrderService JDBC timeout
INC0001002  [BW-JDBC-100014] OrderService JDBC timeout
INC0001003  [BW-JDBC-100014] OrderService JDBC timeout
INC0001004  [BW-JDBC-100014] OrderService JDBC timeout
...
INC0001048  [BW-JDBC-100014] OrderService JDBC timeout
```

**48 tickets for the same slow database connection — generated in under 60 seconds.**

The real problems — a genuinely new REST 500 in PaymentService caused by bad data from S/4HANA, a fraud-screening outage affecting a different integration flow — are buried under this noise.

**This is not a hypothetical.** On active S/4HANA migration projects, customers consistently report that **80%+ of their integration incident queue is redundant** noise from a handful of root causes. The ops team burns hours every morning triaging tickets before they can even see the real issues.

## The Use Case

Our customer is a global manufacturing company in the middle of a **multi-year SAP S/4HANA modernization program**. They are migrating 15 business units from SAP ECC across Order-to-Cash, Procure-to-Pay, and Finance processes. TIBCO BusinessWorks 6 is the integration backbone — 600+ integration applications connecting S/4HANA to downstream systems (payment gateways, logistics platforms, tax engines, data warehouses).

Their integration operations team manages incidents via ServiceNow. As the migration program accelerates, so does the incident volume — and the ops team is drowning:

| Pain Point | Real-World Impact |
|---|---|
| **Duplicate tickets flood the queue** | On-call engineers spend 30+ min triaging noise every morning before finding the 2–3 issues that actually matter |
| **Data quality errors from S/4HANA propagate as cascades** | A single bad IDoc or malformed master data record triggers the same error across 20+ pods — 20 tickets, 1 root cause |
| **Rule-engine dedup is brittle and hard to maintain** | `errorCode == X AND appName == Y` misses cross-flow variants; every new integration requires new rules; nobody owns the rule engine |
| **No context in tickets** | Each ticket is a raw stack trace with no occurrence count, no timeline, no related incidents linked |
| **P1 SLA risk from false merges** | A genuine new incident (e.g., fraud-screening outage) gets silently merged into an existing JDBC ticket because the error code looks similar — and the SLA clock never starts |

**The core ask:** An intelligent triage layer that understands *what the error means in context* — not just string matching — and can decide whether an integration error event is a new problem, a known ongoing issue generating noise, or garbage data that should be discarded entirely.

## Why Flogo + AI Agent?

### The Rule-Engine Trap

Every ops team eventually builds one. It starts sensible:

```python
if event.errorCode == existing.errorCode and event.appName == existing.appName:
    merge()
else:
    create_new()
```

Then reality arrives. The S/4HANA migration adds a new payment integration — new rules. The fraud service gets a URL change — new rules. The error code for a data quality failure looks the same as a network timeout — but they have different owners and different fix paths. Different rules.

After 18 months of a modernization program, the rule engine has 200 conditions, was last touched by someone who left, and nobody trusts it. The ops team starts bypassing it.

The problem is structural: **rule engines encode what engineers knew at the time they wrote the rules.** In a fast-moving S/4HANA migration, the integration landscape changes weekly. The rule engine never catches up.

### What an AI Agent Does Differently

Instead of hard-coding rules, you give the agent:
- A **description of what good triage judgment looks like** (the system prompt)
- **Tools it can call** to look up existing incidents and act on them

The LLM handles the fuzzy, contextual matching that rule engines fail at. Flogo handles the connectivity, retries, rate limiting, and ServiceNow API calls. The result:

```
[Integration Error Event]
         ↓
[Flogo HTTP Trigger]  ← receives structured error payload from any integration app
         ↓
[Validate Payload]    ← deterministic: missing fields → reject immediately, no LLM cost
         ↓
[Agentic AI Activity] ← LLM reasoning loop:
   Step 1: search_incidents(errorCode, appName, last 60 min)
   Step 2: evaluate candidate match quality against confidence rubric
   Step 3: create_incident OR append_occurrence OR reject_bad_data
         ↓
[Return decision JSON] ← { decision, ticketId, confidence, reason }
         ↓
[Audit Log]
```

**The Flogo designer handles all the plumbing.** The triage intelligence lives in the prompt — readable by any engineer, updatable in minutes, no deployment pipeline needed for logic changes.

## How the AI Agent Thinks

This is the heart of the demo. The agent follows a strict 3-step algorithm defined in its system prompt:

### Step 1 — Validate

Before any LLM reasoning, the agent applies **deterministic validation rules**:
- `appName` must be present and non-empty
- `errorCode` must be present and non-empty
- `timestamp` must be present

Failure → `reject_bad_data()` tool called → event logged → `BAD_DATA` decision emitted. No LLM token is spent on garbage.

### Step 2 — Search

The agent calls `search_incidents(errorCode, appName, window_min=60)` — a tool that queries the mock ServiceNow API for **open incidents matching the same error code and app within the last 60 minutes**.

This gives the agent a list of *candidates*. An empty list is unambiguous (guaranteed NEW). A non-empty list requires judgment.

### Step 3 — Confidence Rubric

When candidates exist, the agent scores **match quality using a 3-tier rubric**:

```
HIGH confidence (≥ 0.85)
  Criteria: same processName AND same activityName AND same downstream URL in errorMsg
  Decision: DUPLICATE_OF → append_occurrence()

MEDIUM confidence (0.75 – 0.84)
  Criteria: same processName but different downstream URL,
            OR same downstream URL but different processName
  Decision: DUPLICATE_OF → append_occurrence()

LOW confidence (< 0.75)
  Criteria: different processName AND different downstream URL,
            OR errorMsg indicates intermittent/self-healing behaviour,
            OR severity escalated by 2+ levels vs. candidate
  Decision: NEW ticket (protect SLA) → create_incident()
            + append warning note to original candidate
```

The key insight: **the LLM evaluates semantic similarity, not string equality.** `stripe.example/v1/charge` and `stripe.example/v2/charge` are clearly the same downstream; `fraud-check.internal` and `stripe.example` clearly are not. No regex needed.

### The Low-Confidence Guardrail (Act 5)

When confidence < 0.75, the agent does both:
1. **Creates a new ticket** (`u_decision_source = "agent-low-confidence"`) — SLA is protected.
2. **Appends a note to the original candidate** — `"Agent flagged possible duplicate (low conf=0.62): new ticket INC0001004 opened. Same BW-REST-500 pattern but different process/downstream."`

The on-call engineer sees the link in both tickets and can decide whether to merge. The agent never silently hides a P1.

## Components in Detail

### BW6 Error Simulator (`bw6-error-simulator/simulator.js`)

Three simulation modes:

| Mode | Description | Use |
|---|---|---|
| `--mode=storm` | N events with heavy duplication (3 unique + N-5 duplicates + 2 bad-data) | Act 1 & 2: prove noise reduction |
| `--mode=single` | 1 random event from catalog | Quick sanity check |
| `--mode=edge` | 7 hand-crafted events covering all decision branches | Acts 3–5 |

The `--bypass` flag routes events directly to mock ServiceNow, bypassing the agent — this is used in Act 1 to demonstrate the baseline "noise problem."

**Edge mode event sequence:**

```
Event 1:  JDBC timeout, OrderService             → NEW   (INC0001001)
Event 2:  JDBC timeout, OrderService (again)     → DUP   (merged into INC0001001)
Event 3:  JDBC timeout, PaymentService           → NEW   (INC0001002) ← Act 4 demo
Event 4:  REST-500, PaymentService/ChargeCard    → NEW   (INC0001003)
Event 5:  REST-500, PaymentService/ChargeCard    → DUP   (merged into INC0001003)
Event 6:  REST-500, PaymentService/FraudScreening → LOW-CONF NEW (INC0001004) ← Act 5
Event 7:  Bad data (missing fields)              → REJECTED
```

### Mock ServiceNow (`mock-servicenow/server.js`)

REST API that mirrors real ServiceNow shape so **only the base URL needs to change in production**:

```
POST   /api/now/table/incident            create incident
GET    /api/now/table/incident            search/filter by any field
GET    /api/now/table/incident/:sys_id    get single incident
PATCH  /api/now/table/incident/:sys_id   update: work note, occurrence count, state
GET    /api/now/stats                     demo metrics
POST   /api/now/reset                     clear all state
```

Each incident stores a full `work_notes[]` array — a timestamped log of every agent action on that ticket. This is what the dashboard's expandable timeline renders.

Custom fields (all prefixed `u_`) used by the agent:
- `u_error_code`, `u_app_name`, `u_app_node`, `u_process_name`, `u_activity_name`
- `u_occurrence_count` — how many events were deduplicated into this ticket
- `u_decision_source` — `agent-new` | `agent-low-confidence` | `bypass-baseline`
- `u_agent_confidence` — float 0.0–1.0 from the LLM

### Triage Agent (`bw6-ticket-triage-agent.flogo`)

Flogo application with the following key configuration:

| Property | Value |
|---|---|
| `LLM_Provider` | `OpenAI` (compatible interface) |
| `LLM_Base_URL` | Ollama endpoint (or Azure OpenAI / OpenAI) |
| `LLM_MODEL` | `deepseek-v3.1:671b` (or `gpt-4o`, `claude-3-5-sonnet`) |
| `CONFIDENCE_FLOOR` | `0.75` |
| `rateLimit` | `1000` requests/min |

The agent exposes four MCP tools via the `bw6-sn-mcp-wrapper.flogo` sidecar:
- `search_incidents` → `GET /api/now/table/incident?...`
- `create_incident` → `POST /api/now/table/incident`
- `append_occurrence` → `PATCH /api/now/table/incident/:sys_id`
- `reject_bad_data` → `POST /api/now/reject`

### Dashboard (`dashboard/index.html`)

Static HTML page that polls `http://localhost:8081/api/now/table/incident?sysparm_limit=50` every 2 seconds and renders a live table:

| Column | Description |
|---|---|
| Incident # | ServiceNow `INC` number |
| Error Code | BW6 error code (e.g., `BW-JDBC-100014`) |
| App / Process | `appName` and `processName` |
| Occurrences | How many events were merged |
| Confidence | LLM score — highlighted in purple if `< 0.75` |
| Decision Source | `agent-new`, `agent-low-confidence`, or `bypass-baseline` |
| Work Notes | Expandable timeline of every agent action |

## Running the Demo Step by Step

### Prerequisites

- **Node.js** ≥ 18
- **VS Code** with the TIBCO Flogo extension installed
- An **LLM backend**: Ollama with `deepseek-v3.1:671b`, or an OpenAI/Azure OpenAI key
- Ports `8080` and `8081` free

### Setup (one-time)

```bash
cd bw6-ai-triage-demo

# Install simulator and mock-SN deps
(cd mock-servicenow && npm install)
(cd bw6-error-simulator && npm install)
```

Open `bw6-ticket-triage-agent.flogo` in VS Code and set your LLM credentials in the app properties:

```
LLM_Provider   = OpenAI
LLM_Base_URL   = http://localhost:11434/v1   # or https://api.openai.com/v1
API_Key        = <your key>
LLM_MODEL      = deepseek-v3.1:671b         # or gpt-4o
CONFIDENCE_FLOOR = 0.75
```

### Start Services

Open **three terminals**:

```bash
# Terminal 1 — Mock ServiceNow
cd mock-servicenow && npm start
# Expected: [mock-servicenow] listening on http://localhost:8081

# Terminal 2 — Flogo Triage Agent
# Right-click bw6-ticket-triage-agent.flogo in VS Code → "Run Flogo App"
# Expected: Flogo app started on :8080

# Terminal 3 — (used for simulator commands below)
```

Open `dashboard/index.html` in your browser. Keep it visible throughout.

### Act 4 — Sneaky New One

**Talk track:** *"Here is exactly where rule engines fail in a modernization program. Same error code, different integration flow — different root cause, different owner, different fix."*

```bash
curl -s -X POST http://localhost:8081/api/now/reset > /dev/null && echo "Reset"

```

Watch the dashboard. Point out the key moment when **Event 3** arrives:
- Events 1–2: `BW-JDBC-100014` on `OrderService` (Order-to-Cash flow) → NEW + DUP (merged)
- **Event 3**: `BW-JDBC-100014` on `PaymentService` (Procure-to-Pay flow) → **NEW** (separate incident)

A naive rule engine keyed on `errorCode` alone would merge these — hiding a potentially independent database problem in the payment flow. The agent sees *different appName, different processName, different integration context* and correctly creates `INC0001002`. Two flows, two owners, two investigations.

---

### Act 5 — Human-in-the-Loop Guardrail 

Still in the edge run, watch events 4–6:

- Event 4: `BW-REST-500`, `PaymentService/ChargeCard`, downstream `stripe.example` → **NEW** (`INC0001003`)
- Event 5: Same → **DUP** (merged into `INC0001003`)
- **Event 2**: `BW-REST-500`, `PaymentService/FraudScreening`, downstream `fraud-check.internal`

Event 6 hits both LOW-confidence criteria:
- Different `processName` (`FraudScreening` vs `ChargeCard`)
- Different downstream service (`fraud-check.internal` vs `stripe.example`) and different processName

**Expected result:** The agent creates `INC0001004` with `u_decision_source=agent-low-confidence` AND appends a warning note on `INC0001003`.

In the dashboard, expand `INC0001003`'s work notes. You'll see a note like:
> *"Agent flagged possible duplicate (low conf=0.62): new ticket INC0001004 opened. Same BW-REST-500 pattern but different process/downstream — creating new ticket to protect SLA."*

The `lowc` purple pill next to `INC0001004`'s confidence score makes this stand out visually.

**Talk track:** *"During an S/4HANA migration, erring on the side of a new ticket is the right call. A silently-merged P1 — say, a fraud-screening outage misclassified as a payment timeout — could mean a missed SLA and real business impact. The agent surfaces the ambiguity instead of hiding it. The on-call engineer gets both tickets, sees the cross-reference note, and makes the final judgment."*

## Interpreting the Results

### Decision Sources in the Dashboard

| `u_decision_source` | Meaning |
|---|---|
| `agent-new` | LLM was confident this is a new, unique incident |
| `agent-low-confidence` | LLM was uncertain — new ticket opened to protect SLA, warning left on candidate |
| `bypass-baseline` | Bypassed the agent (Act 1 baseline) |

### Work Note Color Legend

| Color | Meaning |
|---|---|
| 🟢 Green dot | Normal occurrence note (DUP merged) |
| 🟡 Amber dot | Duplicate-merge note |
| 🟣 Purple dot | Low-confidence warning note |
| 🔴 Red dot | Bad-data rejection note |

### The `lowc` Pill

A purple `lowc` badge on any incident means `confidence < 0.75`. These are the tickets that warrant a human review — the agent has done its best but is not certain.

## Production Path

This demo is production-shaped by design. To go from demo to real environment:

### What to Swap

| Demo Component | Production Equivalent |
|---|---|
| Mock ServiceNow (`localhost:8081`) | Real ServiceNow instance URL (update base URL in `bw6-sn-mcp-wrapper.flogo`) |
| `bw6-error-simulator` | Real integration app fault handler — a BW6 process fault handler or global exception handler that POSTs the error event to the triage agent |
| Ollama / local LLM | Azure OpenAI, OpenAI GPT-4o, or Claude via Flogo Agentic AI connector — change 3 app properties |
| In-memory store in mock-SN | Real ServiceNow `sys_id` returned from real API — no code changes needed |

### What Stays the Same

- The `bw6-ticket-triage-agent.flogo` **Flogo app structure** — only the base URL and API key change
- The **system prompt and confidence rubric** — this encodes your team's triage knowledge
- The **four MCP tools** (`search_incidents`, `create_incident`, `append_occurrence`, `reject_bad_data`) — these are thin wrappers over standard ServiceNow REST; they work as-is
- The **dashboard** — point it at your real ServiceNow read endpoint

### Tuning the Agent

| Parameter | Where | Effect |
|---|---|---|
| `CONFIDENCE_FLOOR` | App property | Raise to 0.85 for stricter dedup; lower to 0.60 to merge more aggressively |
| `window_min=60` | System prompt | Change the lookback window for candidate search |
| HIGH/MEDIUM/LOW rubric | System prompt | Add domain-specific signals (e.g., `u_app_node` pod index, error frequency) |
| `LLM_MODEL` | App property | Swap models without touching any other code |

### Scaling

The Flogo agent is stateless. Deploy multiple instances behind a load balancer. The only shared state is the ServiceNow incident store (already the single source of truth in real deployments).
